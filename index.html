☁️ Cloud Computing Basics Fundamentals
▾

Cloud = infrastructure + automation + orchestration, billed by usage (OpEx) instead of upfront purchase (CapEx).

Key Points
  • On-premises / Data center: you own and manage all hardware — CapEx model (pay upfront)
  • Private cloud: your own rack with automation & orchestration tools — still CapEx
  • Public cloud: shared resources owned by provider (AWS) — OpEx model (pay-as-you-go)
  • IaaS / PaaS / SaaS: layers of how much AWS manages vs you manage
  • XaaS: "Anything as a Service" — storage, network, hardware all rentable

🔐 IAM — Identity & Access Management Security
▾

IAM controls who can access AWS and what they are allowed to do — it is global across all regions.

Key Points
  • Root user: the account owner — full access, should be locked away after initial setup
  • IAM user: a named identity with specific permissions you define
  • 4 ways to access AWS: Console (password), Programmatic (access key + secret key), IAM Roles (temporary sessions), APIs
  • Permissions use explicit permit / implicit deny: no matching rule = denied
  • Policies are written in JSON format
  • IAM is a global service — not locked to any region, unlike VPC

🏙️ VPC — Virtual Private Cloud Networking
▾

A VPC is your own isolated virtual data center inside a specific AWS region.

Key Points
  • Isolated from other VPCs and other AWS customers
  • Confined to one region — unlike IAM which is global
  • Connect to on-premises data center via VGW (Virtual Gateway)
  • AWS Direct Connect: dedicated physical line → high performance
  • VPN: encrypted tunnel over internet → high security
  • Subnets live inside a VPC within a specific AZ (Availability Zone)

🖥️ EC2 — Elastic Compute Cloud Compute
▾

EC2 gives you a resizable virtual machine in the cloud — your "server" that you can start, stop, and resize.

Key Points
  • Shared host: your VM shares the physical server with other AWS customers (cheapest)
  • Dedicated host: entire physical server is yours alone (for licensing/compliance)
  • Connect to Linux instances via SSH (Secure Shell)
  • EC2 lives inside a subnet, connected via ENI (Elastic Network Interface)
  • Protected by a Security Group acting as a virtual firewall

🧱 Security Groups & NACLs Security
▾

Two layers of firewall — Security Groups protect individual instances, NACLs protect entire subnets.

Key Points
  • Security groups are stateful: if inbound is allowed, the outbound response is automatically allowed
  • Security groups have permit rules only — no explicit deny; no matching rule = implicit deny
  • Inbound rules use source IP + destination port
  • NACLs operate at the subnet level, applied at the implied router
  • NACLs are stateless: you must explicitly allow both inbound AND outbound
  • NACLs support both permit and deny rules

💾 Block Storage — EBS & Instance Store Storage
▾

Two types of disk for EC2 — EBS is persistent and detachable, Instance Store is faster but lost on stop/terminate.

Key Points
  • EBS: volume lives outside the physical server, within the same AZ, connected via ENI
  • EBS holds the operating system — persists when instance stops
  • Instance Store: volume is inside the same physical server — extremely fast
  • Instance Store is ephemeral (volatile) — data is lost when instance stops or terminates

🪣 S3 — Simple Storage Service Storage
▾

S3 is object storage — store any file via a web interface in containers called Buckets, outside your VPC.

Key Points
  • Storage type is object storage (not block) — stores files as whole objects with metadata
  • Storage container is called a Bucket
  • Data exists within one region but outside the VPC
  • Accessed via a web service interface (HTTP/HTTPS)
  • Permissions use explicit (rule exists in policy) and implicit (no rule = deny) logic

🌐 CloudFront & Route 53 Networking
▾

CloudFront is AWS's CDN that caches content at edge locations globally. Route 53 is AWS's DNS service.

Key Points
  • CloudFront connects edge locations to S3 buckets via high-speed links
  • Edge locations cache content — no full data duplication needed
  • Using an edge location is cheaper and faster than users hitting the S3 bucket directly
  • Route 53 is the DNS (Domain Name System) service for AWS — translates domain names to IPs

🔑 KMS — Key Management Service Security
▾

KMS manages encryption keys for your AWS data — it is a regional service.

Key Points
  • KMS is responsible for encryption across AWS services
  • It is a regional service — keys in one region cannot be used in another
  • Primary resource = CMK (Customer Master Key)
  • AWS-managed CMK: created, managed, and used by AWS on your behalf
  • Customer-managed CMK: you create, manage, and control it yourself

🗄️ RDS & OLTP vs OLAP Databases
▾

RDS is AWS's managed relational database service — designed for OLTP (transactional) workloads, not analytics.

Key Points
  • RDS lives inside your VPC — best placed in a private subnet
  • Can be fully managed by AWS or self-managed on EC2
  • Can run standalone or with a standby replica in another AZ for high availability
  • OLTP: small fast queries, CRUD operations, fixed schema (RDS fits here)
  • OLAP: complex analytical/reporting queries — use Redshift instead

📊 CloudWatch Monitoring
▾

CloudWatch is the central monitoring hub in AWS — it collects metrics and performance data for your resources.

Key Points
  • CloudWatch is the center of monitoring and visibility in AWS
  • It is a metric data repository — collects numbers (CPU %, network in/out, disk, etc.)
  • Focuses on one resource at a time — you define what to monitor
  • Can trigger alarms and auto-scaling actions based on thresholds

📋 CloudTrail Monitoring
▾

CloudTrail logs every API call made in your AWS account — it's your audit trail for "who did what."

Key Points
  • Records every API call made — at the service level
  • Logs can be archived or analyzed by storing them in an S3 bucket via CloudTrail Trails
  • Default scope: all regions. Can be limited to one region via CLI only
  • Supports file integrity validation — detects if logs have been tampered with
  • Different from CloudWatch — CloudTrail = "who did it", CloudWatch = "how is it performing"
